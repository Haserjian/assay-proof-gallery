# Repo State Policy

**What belongs in this repo, what does not, and why.**

---

## Repo content (must be versioned)

| Path | What it is |
|------|-----------|
| `gallery/*/` | Scenario artifacts: proof packs, tamper packs, README, verify.sh |
| `scenarios.json` | Gallery contract: scenario IDs, expected exit codes |
| `scripts/` | Gallery build and check scripts |
| `docs/`, `*.html`, `*.js` | Browser verifier surface |
| `.gitignore` | This policy's enforcement layer |
| `GALLERY_VOCABULARY.md` | Canonical term definitions |
| `REPO_STATE_POLICY.md` | This file |

## Generated local state (gitignored — do not commit)

| Path | Generated by | Why ignored |
|------|-------------|-------------|
| `receipts/` | Session receipt files from agent work | Not gallery artifacts; agent-local session state |
| `.assay/` | `assay vendorq verify` and other CLI commands writing to CWD | Tool runtime output; timestamped reports for temp packs that no longer exist |
| `__pycache__/`, `*.pyc` | Python interpreter | Standard |
| `.venv/` | `python -m venv` | Local virtual environment |
| `site/` | MkDocs build | Regenerable |
| `*.egg-info/` | `pip install -e .` | Build artifact |

## Must never be committed

- Private keys, signing keys, or key material of any kind
- Credentials, tokens, or secrets
- Verification reports tied to temp or non-reproducible paths (the CWD-output trap: when running `assay` commands from the repo root, CLI output may land here — check `git status` before committing)

## The CWD output trap

Several `assay` CLI commands write output to the working directory by default
(`assay vendorq verify`, `assay run`, etc.). When run from the repo root, this
creates files under repo scope that are not repo content. Rule:

> Before `git add .` or `git commit -am`, run `git status --short` and
> explain every line. If a file cannot be explained as intentional repo
> content, it does not get committed.

## Branch protection note

`main` is protected: the default rule requires changes to come through a PR.
The repo owner holds a bypass actor exemption, so direct pushes are also
allowed in current practice. This is a normal solo/bootstrap repo setup —
two valid paths exist (PR and direct push), not a broken rule.

**Current mode:** bootstrap — owner direct-push is intentional while the repo
is single-contributor. If this becomes a shared or audited repo, removing the
bypass exemption and enforcing the PR path exclusively is the right move at
that point.
