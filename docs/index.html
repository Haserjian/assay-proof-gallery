Exit code contract

Exit 1 = authentic evidence of failure, not corruption. The pack is valid. The failure is the proof.

ExitMeaningWhat it tells an auditor
0 PASS Integrity intact, all declared claims pass.
1 HONEST FAIL Pack intact — the evidence is authentic and proves a policy violation.
Exit 1 is not corruption. It means the evidence survived and it shows the system missed its declared standard. The failure cannot be rewritten after signing.
2 TAMPERED Pack bytes were altered after signing — integrity broken.
Detectable offline: SHA-256 mismatch against the manifest, which the Ed25519 signature still covers.
Why this matters

Use this when a customer, auditor, or regulator asks: prove the controls actually ran.

Failure can't be hidden

A policy violation sealed at runtime cannot be edited out after the fact. The pack either passes verification or it doesn't — and exit 1 is proof the failure happened and is authentic.

No platform access needed

Auditors verify packs on their own machine with the public CLI. No login, no API key, no trust in the vendor's live infrastructure. The artifact is the proof.

Tampering is detectable

Change one byte in the receipt file. The verifier catches it. The manifest and signature don't need to be touched — the hash mismatch is enough.

What's inside a proof pack
pack_manifest.json ← SHA-256 hashes of every file + Ed25519 signature over the manifest
pack_signature.sig ← raw signature bytes, verified against the public key embedded in the manifest
receipt_pack.jsonl ← execution receipts (model calls, guardian verdicts, capability uses) — one per line
verify_report.json ← machine-readable result: receipt_integrity, claim_check, errors
verify_transcript.md ← human-readable transcript — forwardable to auditors as-is
